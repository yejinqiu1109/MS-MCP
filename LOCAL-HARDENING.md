# Local hardened deployment

MS-MCP is designed for a single-user Windows workstation. Keep the MCP stdio server and Dashboard local, use a dedicated writable workspace, and retain the fail-closed defaults generated by `Configure-MS-MCP.bat`.

## Required boundaries

- Keep `MS_MCP_ALLOW_ARBITRARY_SCRIPT=0` unless every caller and prompt is fully trusted.
- Keep `MS_MCP_ALLOW_EXTERNAL_INPUTS=0`; copy approved inputs into the workspace instead.
- Limit CIF hosts to explicitly trusted HTTPS sources and retain a reasonable byte limit.
- Keep Dashboard bound to loopback. Enabling Dashboard writes requires a random token of at least 24 characters.
- Do not store server passwords. Remote monitoring uses a dedicated SSH key and should have only the remote permissions it needs.
- Do not upload `config/ms-mcp.local.*`, `.env`, workspaces, calculation results, logs, private keys, license files, or proprietary structures.

## Path changes

Absolute paths are intentionally centralized in the generated local BAT and Codex TOML. After moving the repository, Node.js, Materials Studio, workspace, model directory, or remote Gateway, re-run `Configure-MS-MCP.bat`, replace the Codex MCP block, and re-run `Test-MS-MCP.bat`.

## Verification

Run `Test-MS-MCP.bat` after installation and after every path or runtime change. Real compute workflows require separate validation with a small authorized structure because the smoke suite does not consume a license or submit production calculations.
