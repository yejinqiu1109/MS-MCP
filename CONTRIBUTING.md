# Contributing

1. Create a focused branch and avoid committing generated or proprietary data.
2. Run `npm ci`, `npm run check`, and the applicable smoke tests.
3. Describe Materials Studio and Node versions used for validation without publishing license or server details.
4. Keep new paths configurable; do not add author-specific drive letters, usernames, IP addresses, or Gateway directories.
5. Preserve fail-closed defaults for arbitrary scripts, external inputs, Dashboard writes, and network sources.

Pull requests should explain behavior changes, safety impact, test evidence, and any installation/configuration migration required.
